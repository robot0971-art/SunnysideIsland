# 작업 완료 확인 가이드

> AI가 작업을 완료한 후 사용자가 확인해야 할 항목들입니다.

---

## 🔍 확인 프로세스

```
AI 작업 완료 → 사용자 확인 → 피드백 → 다음 단계
     ↑___________________________________|
```

---

## ✅ 공통 확인 항목 (모든 작업)

### 1. 문서 확인
- [ ] 생성된 문서가 `docs/impl/phases/M{X}/`에 있는지 확인
- [ ] 문서명이 규칙에 맞는지 확인 (`RESEARCH_M1_xxx.md`, `PLAN_M1_xxx.md`)
- [ ] 문서 내용이 템플릿을 따르는지 확인
- [ ] 링크/경로가 올바른지 확인

### 2. 코드 확인
- [ ] 코드가 `Assets/Scripts/{System}/`에 있는지 확인
- [ ] AGENTS.md 코딩 규칙 준수 여부 확인
- [ ] DI 패턴 적용 확인 (`[Inject]`, `Register` 등)
- [ ] EventBus 이벤트 사용 확인
- [ ] 주석/XML 문서화 확인

### 3. 파일 구조 확인
- [ ] 폴더 구조가 프로젝트 규칙에 맞는지 확인
- [ ] 네이밍 컨벤션 준수 확인 (PascalCase, camelCase 등)
- [ ] 네임스페이스가 올바른지 확인

## 📋 Phase별 확인 항목

### RESEARCH 문서 완료 시

```markdown
## ✅ 확인 체크리스트

### 기획 문서 검토
- [ ] 관련 기획 문서(GDD, 구현계획)를 참고했는가?
- [ ] 요구사항이 모두 정리되었는가?
- [ ] 인터페이스 정의가 초안이나 결정사항이 포함되었는가?

### 기존 코드 분석
- [ ] 유사 시스템을 검색했는가? (grep 결과 포함)
- [ ] 관련 클래스를 분석했는가?
- [ ] 재사용 가능한 코드를 식별했는가?
- [ ] 데이터 클래스(ScriptableObject)를 분석했는가?

### 의존성 분석
- [ ] 의존하는 시스템을 목록화했는가?
- [ ] 의존되는 시스템을 목록화했는가?
- [ ] 의존성 그래프를 그렸는가?

### 리스크 분석
- [ ] 기술적 리스크를 식별했는가?
- [ ] 일정적 리스크를 식별했는가?
- [ ] 대응 방안을 제시했는가?

### 구현 방향
- [ ] 아키텍처 결정 사항이 명확한가?
- [ ] 선택지와 선택 이유를 제시했는가?
- [ ] 클래스 설계 초안이 포함되었는가?
- [ ] 파일 구조를 제시했는가?

### 구현 계획
- [ ] 구현 단계가 시간 순서대로 정리되었는가?
- [ ] 테스트 계획이 포함되었는가?
- [ ] 완료 조건이 명확한가?
```

**확인 후**: "리서치 완료, PLAN 문서 작성으로 진행" 또는 "수정 필요: {내용}"

## 🔄 이미 구현된 코드 처리

### 확인 절차

```
1. AI가 기존 코드 검색
   ↓
2. AI가 코드 분석 및 상태 평가
   ↓
3. 사용자에게 보고
   - ✅ 완료: 그대로 사용
   - ⚠️ 부분 수정: 수정 필요 부분 명시
   - ❌ 재구현: 이유 명시
   ↓
4. 사용자 확인
   ↓
5. 필요시 수정 또는 넘어가기
```

### 상태 분류

| 상태 | 표시 | 처리 | 예시 |
|------|------|------|------|
| 완료 | ✅ | 그대로 사용, 체크만 하고 넘어감 | DI Container, EventBus |
| 부분 수정 | ⚠️ | 필요한 부분만 수정 | 기능 보강, 버그 수정 |
| 재구현 | ❌ | 새로 구현 | 설계 변경, 요구사항 변경 |
| 없음 | ⏳ | 신규 구현 | 새로운 시스템 |

### 시나리오 1: DI Container (이미 구현)

**AI 보고**:
```
DI Container 분석 결과:
- 위치: Assets/Scripts/DI/DIContainer.cs
- 상태: ✅ 완료
- 기능: 싱글톤, 인터페이스 등록, 인스턴스 등록, 자동 주입 모두 구현
- 테스트: 필요시 보강 가능

결론: 현재 구현을 그대로 사용합니다.
추가 작업: 테스트 코드 작성 (선택)
```

**사용자 확인**:
- [ ] 코드 위치 확인
- [ ] 기능 확인
- [ ] "확인 완료, 다음으로 진행" 또는 "테스트 코드 추가해줘"

**AI 보고**:

```
TimeManager 분석 결과:
- 기존 코드: TimeOfDayData.cs 존재 (데이터만)
- 구현 필요: TimeManager 클래스 전체
- 의존성: EventBus, DI Container (✅ 완료)

결론: 신규 구현이 필요합니다.
```

**사용자 확인**:
- [ ] 리서치 문서 확인

- [ ] "승인, 구현 진행" 또는 "수정 필요"

---

### 시나리오 3: EventBus (이미 구현, 부분 보강 필요)

**AI 보고**:
```
EventBus 분석 결과:
- 위치: Assets/Scripts/EventBus.cs
- 상태: ⚠️ 부분 수정 필요
- 현재 기능: 기본 발행/구독 구현
- 보강 필요:
  1. 코어 이벤트 정의 (GameStartedEvent 등)
  2. 시간 이벤트 정의 (TimeChangedEvent 등)
  3. 플레이어 이벤트 정의

결론: 기본 클래스는 유지, 이벤트 클래스들만 추가 구현
```

**사용자 확인**:
- [ ] 기존 코드 확인
- [ ] 보강 필요 사항 확인
- [ ] "승인, 보강 구현 진행" 또는 "다른 방식으로 해줘"
